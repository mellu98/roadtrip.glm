================================================================================
2. PRINCIPI GUIDANTI
================================================================================

PRINCIPIO 1: RESTARE CLIENT-SIDE
  Non aggiungere un backend. La forza dell'app è essere deployabile come
  static site su GitHub Pages, Vercel, Netlify, ecc. Ogni feature deve
  funzionare senza server.

PRINCIPIO 2: AI-FIRST
  L'IA è il cuore. Ogni miglioramento deve POTENZIARE l'esperienza AI,
  non sostituirla. TREK è manuale, noi siamo AI-first. Questo è il
  nostro differenziale competitivo.

PRINCIPIO 3: PROGRESSIVE ENHANCEMENT
  Ogni feature deve funzionare anche senza dati aggiuntivi. Se Open-Meteo
  non risponde, l'app continua a funzionare senza meteo. Se Leaflet non
  carica, i link a Google Maps restano come fallback.

PRINCIPIO 4: MOBILE-FIRST
  L'app è PWA. Ogni feature è pensata prima per mobile, poi per desktop.

PRINCIPIO 5: PERFORMANCE
  Lazy loading per tutte le nuove dipendenze (Leaflet, Chart.js, ecc.).
  Non gonfiare il bundle iniziale.

9. RISCHI E MITIGAZIONI
================================================================================

RISCHIO 1: BUNDLE SIZE
  Problema: Leaflet, Chart.js, @dnd-kit aumentano il bundle
  Mitigazione: lazy loading di tutti i moduli pesanti
    const MapView = React.lazy(() => import('./components/MapView'))
    const BudgetTracker = React.lazy(() => import('./components/BudgetTracker'))
  Obiettivo: bundle iniziale < 200KB gzip

RISCHIO 2: RATE LIMITING NOMINATIM
  Problema: Nominatim ha rate limit 1 req/s per policy
  Mitigazione: debounce 300ms + cache aggressiva (24h) + fallback input libero

RISCHIO 3: LIMITI URL CONDIVISIONE
  Problema: Itinerari lunghi generano URL troppo grandi (>2000 char)
  Mitigazione: compressione lz-string + warning se il viaggio è troppo lungo
    per essere condiviso via URL + alternativa export JSON

RISCHIO 4: COSTI API AI PER RIGENERAZIONE
  Problema: Rigenerazione selettiva = più chiamate API = più costi
  Mitigazione: mostrare stimazione token prima di rigenerare
    + opzione "rigenera tutto" come alternativa più economica

RISCHIO 5: localStorage QUOTA
  Problema: Con spese, packing state, meteo cache, ecc. localStorage si riempie
  Mitigazione: IndexedDB per dati voluminosi (spese, cache meteo)
    + cleanup automatico cache scaduta + warning se vicino al limite

RISCHIO 6: COMPATIBILITÀ LEAFLET SSR/SSG
  Problema: Leaflet non funziona in SSR (ma la nostra app è SPA pura, ok)
  Mitigazione: nessuna — siamo SPA pura, nessun rischio

RISCHIO 7: i18n È RUTTURA E DISTURBO
  Problema: Sostituire tutte le stringhe è lungo e incline a errori
  Mitigazione: fare per ultimo + automatizzare estrazione con script
    + coprire solo it/en (non 8 lingue come TREK)
